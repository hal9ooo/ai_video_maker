## 1. Project Context

The project is an **automated music-video pipeline**. The relevant scripts are:

| File                            | Role                                                                                                |
| ------------------------------- | --------------------------------------------------------------------------------------------------- |
| `auto_lyrics_subtitles_groq.py` | Generates `.srt`, `.ass`, and word-timing `.json` from a vocal stem + lyrics using Groq Whisper API |
| `automated_music_video.py`      | Assembles video clips into a final video with audio                                                 |
| **`add_subtitles_to_video.py`** | Burns styled subtitles into the assembled video ← **this is the file under development**            |

### Directory layout

```
input_automated/
  audio/Relive.wav              ← full song (vocals + instruments)
  vocal_stems/vocal_stem.wav    ← isolated vocals only

  subtitles/
    vocal_stem_lyrics.srt       ← SRT generated by auto_lyrics_subtitles_groq.py

output_automated/

  output_karaoke.mp4            ← various test outputs (can be deleted)
  ...
```

## 2. What `add_subtitles_to_video.py` Does (Current State)

### Working features ✅

1. **4 style presets** (`--style`):
   - `neon_night` – Helvetica Neue Bold, cyan `#7DEBFF`, glow on dark
   - `warm_cinematic` – Georgia Bold, amber `#F6C374`
   - `bold_impact` – Impact, white `#FFFFFF`, thick black outline
   - `clean_white` – Arial, white `#F6F7FA`, minimal

2. **Social-media safe-zone positioning** (`--platform`):
   - `instagram_reels` → 22% bottom reserved (default)
   - `tiktok` → 20%
   - `youtube_shorts` → 18%
   - `generic` → 15%
   - Margin is scaled to actual video height (calibrated at 1920px)

3. **Text wrapping**: calculates max chars/line from `font_size × 0.55 × usable_width`, wraps to max 2 lines

4. **ASS generation from SRT**: reads `.srt`, generates a styled `.ass` with correct timing, wrapping, and margin

5. **FFmpeg burning**: uses `ass=` filter via `ffmpeg-full` (which has libass)

6. **FFmpeg binary auto-detection**: prefers `/opt/homebrew/opt/ffmpeg-full/bin/ffmpeg` (has libass), falls back to system `ffmpeg`

7. **Dry-run mode** (`--dry-run --keep-ass`): generates ASS without encoding

8. **Style overrides**: `--font`, `--font-size`, `--primary-color`, `--outline-color`, `--bold`, `--italic`, `--outline-width`, `--shadow-depth`

9. **Subtitle offset** (`--subtitle-offset N`): shifts all subtitle timestamps by N seconds

10. **Auto-offset** (`--auto-offset`): detects vocal stem silence vs video audio start to compute offset automatically

### Non-karaoke mode works correctly ✅

Running without `--karaoke` uses the SRT line timings directly and produces correct output:

```bash
python3 add_subtitles_to_video.py --style neon_night --use-srt
```

This was tested and confirmed working.

## 3. What's Broken ❌

### Karaoke mode is completely broken

The `--karaoke` flag activates word-by-word highlight using `{\kf}` ASS tags and the word-timing JSON. **The subtitles are not aligned correctly.**

#### Root cause analysis

The word-timing JSON (`vocal_stem_word_timing.json`) has **bad timing data from Groq Whisper**:

```json
{
  "word": "Found",
  "start": 0.02,     ← WRONG: should be ~14.8s (vocal stem has 14.8s of silence before first vocal)
  "end": 15.08,      ← This makes "Found" a 15-second word!
  "confidence": null
}
```

The vocal stem (`vocal_stem.wav`) has:

- **Silence from 0 to 14.79s** (instrumental intro)
- First vocal at **14.79s**

But Groq placed "Found" at 0.02s. This means:

- The SRT entry `00:00:00,020 --> 00:00:18,140` is also wrong (starts 14.8s too early)
- The word timing JSON has the first word spanning 15 seconds

#### What was tried

1. **Auto-offset** (`--auto-offset`): Detects that the vocal stem has 14.8s of silence and the video audio starts at 0.66s. Computes offset = `14.8 - 0.66 = +14.14s`. This shifts all subtitles forward by 14.14s. **The user said this made things worse** – the subtitle appeared at 14s and "stayed on after the next phrase."

2. **Proportional word distribution**: When the first word takes >80% of the line duration, the karaoke mode falls back to distributing word highlights proportionally by character length. **The user said this is "completely broken – all subtitles are not aligned."**

3. **No offset** (offset=0): The user confirmed the SRT timing is correct as-is (offset should be 0). The non-karaoke mode works fine with offset=0.

#### The real problem

The karaoke mode's `_proportional_word_timings()` function and the word-matching logic in `build_karaoke_ass()` are not working correctly. The function:

- Uses a wider search window (`search_margin = max(2.0, (entry.end - entry.start) * 0.5)`) which may be pulling in words from adjacent lines
- The proportional fallback distributes time evenly but doesn't account for the actual word timing within the SRT line window
- The word matching between SRT text and JSON words may be misaligning

## 4. Key Technical Details

### FFmpeg on macOS

- Standard Homebrew `ffmpeg` does **NOT** include libass → `ass` filter is missing
- Must use `ffmpeg-full`: `brew install ffmpeg-full`
- Installed at: `/opt/homebrew/opt/ffmpeg-full/bin/ffmpeg` (keg-only, not in PATH)
- The script auto-detects this path via `_find_ffmpeg()`

### ASS filter path escaping

FFmpeg's `ass=` filter has issues with long paths on macOS. The script copies the ASS file to `/tmp/ffmpeg_subs_burn.ass` before passing it to FFmpeg.

## 5. What Needs to Be Done Next

### Priority 1: Fix karaoke mode

The karaoke word-highlight mode needs to be rewritten. The approach should be:

1. **Use the SRT timing as the source of truth** for when each subtitle line appears/disappears (the user confirmed SRT timing is correct with offset=0)

2. **For word-level timing within each line**, use the word-timing JSON but:
   - Only match words that actually belong to the current SRT line
   - Clamp all word timings to fit within the SRT line's `[start, end]` window
   - If a word's timing falls outside the line window, redistribute proportionally
   - Handle the case where Groq gave wrong absolute times but correct relative ordering

3. **Consider a simpler approach**: Instead of trying to use the JSON word timings (which are unreliable), just split each SRT line's duration proportionally among its words based on character/syllable count. This gives a smooth karaoke sweep that's always in sync with the SRT timing.

### Priority 2: Add VideoToolbox hardware acceleration

The Mac has `h264_videotoolbox` available in ffmpeg-full. Add a `--hwaccel` flag to use it instead of `libx264` for faster encoding. The command would change from:

```
-c:v libx264 -crf 18 -preset slow
```

to:

```
-c:v h264_videotoolbox -q:v 65
```

(VideoToolbox uses `-q:v` instead of `-crf`, range 1-100 where lower = better quality)

### Priority 3: Clean up test outputs

The `output_automated/` directory has many test files that should be cleaned up:

- `output_karaoke.mp4`
- `output_karaoke_nooffset.mp4`
- `output_karaoke_synced.mp4`
- `output_karaoke_v2.mp4`
- `output_subtitled.mp4`
- `output_final.mp4`

Only `output.mp4` (the source video) should be kept.

### `auto_lyrics_subtitles_groq.py` – Generates the subtitle files

This script uses Groq Whisper API to transcribe the vocal stem and align it with known lyrics. It produces:

- `.srt` – standard subtitle file
- `.ass` – styled ASS subtitle file (with a "Mood" style preset)
- `.json` – per-word timing data

The ASS file it generates has its own style (Segoe UI Semibold, `#FFD47A`, MarginV=300). When `add_subtitles_to_video.py` uses `--use-srt`, it regenerates the ASS with its own style presets.

### Word timing JSON format

```json
{
  "words": [
    {
      "word": "Found",
      "start": 0.02,
      "end": 15.08,
      "confidence": null
    },
    ...
  ]
}
```

### SRT format

```
1
00:00:00,020 --> 00:00:18,140
Found your jacket in the closet

2
00:00:21,680 --> 00:00:24,430
Ticket stubs inside the sleeve
```
